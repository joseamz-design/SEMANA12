Respaldo Completo Institucional de GradosTitulos

▼

📋 Enunciado

La Oficina de TI requiere una política de respaldo completo diario para GradosTitulos que almacena información crítica de estudiantes, trámites, evaluaciones, resoluciones y diplomas. Crear un procedimiento que genere el respaldo y registre la fecha de ejecución.

💻 Script T-SQL

-- Procedimiento de respaldo completo con fecha dinámica
CREATE OR ALTER PROCEDURE usp_BackupCompleto_GradosTitulos
AS
BEGIN
    DECLARE @ruta    NVARCHAR(500)
    DECLARE @fecha   NVARCHAR(20)
    DECLARE @sql     NVARCHAR(1000)

    -- Nombre de archivo con fecha/hora
    SET @fecha = CONVERT(NVARCHAR, GETDATE(), 112)
               + '_'
               + REPLACE(CONVERT(NVARCHAR,GETDATE(),108),':','')

    SET @ruta = 'C:\Backups\GradosTitulos\Full\GT_Full_'
              + @fecha + '.bak'

    -- Ejecutar backup con compresión
    SET @sql = 'BACKUP DATABASE GradosTitulos
        TO DISK = ''' + @ruta + '''
        WITH COMPRESSION, CHECKSUM,
             NAME = ''GradosTitulos Full '' + @fecha + ''',
             DESCRIPTION = ''Backup completo institucional'''

    EXEC sp_executesql @sql

    -- Registrar en tabla de control
    INSERT INTO dbo.HistorialBackup
        (TipoBackup, FechaEjecucion, RutaArchivo, Estado)
    VALUES
        ('FULL', GETDATE(), @ruta, 'COMPLETADO')

    PRINT 'Backup completo generado: ' + @ruta
END

🔬 Justificación Técnica

  • WITH COMPRESSION reduce hasta 60-70% el tamaño del archivo .bak.
  • WITH CHECKSUM valida la integridad durante la escritura del backup.
  • La tabla HistorialBackup permite auditoría y monitoreo automatizado.

💡 Buenas Prácticas

  • Programar como SQL Agent Job, ejecutado a las 2:00 AM domingo.
  • Copiar el archivo .bak a ubicación externa (NAS o nube) inmediatamente.
  • Verificar el backup con RESTORE VERIFYONLY tras cada ejecución.

Backups Diferenciales para Trámites Académicos

▼

📋 Enunciado

La universidad necesita reducir tiempos de respaldo combinando respaldos completos semanales y diferenciales diarios para los esquemas Tramite y Documento.

💻 Script T-SQL

-- Backup diferencial diario
DECLARE @ruta NVARCHAR(500)
SET @ruta = 'C:\Backups\GradosTitulos\Diff\GT_Diff_'
           + CONVERT(NVARCHAR, GETDATE(), 112) + '.bak'

BACKUP DATABASE GradosTitulos
    TO DISK = @ruta
    WITH DIFFERENTIAL,
         COMPRESSION,
         CHECKSUM,
         NAME = 'GT Diferencial '
              + CONVERT(VARCHAR,GETDATE(),103),
         DESCRIPTION = 'Backup diferencial GradosTitulos';

-- Verificar el diferencial generado
RESTORE VERIFYONLY
    FROM DISK = @ruta
    WITH CHECKSUM;

-- Cuánto cambió respecto al Full
SELECT backup_size/1024/1024 AS size_mb,
       backup_finish_date, type
FROM   msdb.dbo.backupset
WHERE  database_name = 'GradosTitulos'
ORDER BY backup_finish_date DESC;

🔬 Justificación Técnica

  • El diferencial es acumulativo desde el último FULL, no desde el diferencial anterior.
  • RESTORE VERIFYONLY lee el backup sin restaurarlo, validando su integridad.
  • msdb.dbo.backupset es el catálogo histórico de todos los backups realizados.

💡 Buenas Prácticas

  • Ejecutar diferencial L-S entre las 11:00 PM y 12:00 AM.
  • Mantener mínimo el último FULL + todos los diferenciales hasta ese Full.

Estrategia Integral Full + Differential + Log Backup

▼

📋 Enunciado

GradosTitulos opera 24/7. Diseñar estrategia con respaldo semanal, diferencial diario y log cada 30 minutos.

💻 Script T-SQL

-- Verificar modelo de recuperación (debe ser FULL para logs)
SELECT name, recovery_model_desc
FROM   sys.databases
WHERE  name = 'GradosTitulos';

-- Asegurar modelo FULL
ALTER DATABASE GradosTitulos
    SET RECOVERY FULL;

-- 1. FULL semanal (domingo 2:00 AM)
BACKUP DATABASE GradosTitulos
    TO DISK = 'C:\Backups\Full\GT_Full_Semanal.bak'
    WITH COMPRESSION, CHECKSUM, INIT;

-- 2. DIFERENCIAL diario (lunes a sábado 11 PM)
BACKUP DATABASE GradosTitulos
    TO DISK = 'C:\Backups\Diff\GT_Diff_Diario.bak'
    WITH DIFFERENTIAL, COMPRESSION, CHECKSUM, INIT;

-- 3. LOG cada 30 minutos (mediante Agent Job)
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\GT_Log_'
             + CONVERT(VARCHAR,GETDATE(),112)
             + REPLACE(CONVERT(VARCHAR,GETDATE(),108),':','')
             + '.trn'
    WITH COMPRESSION, CHECKSUM;

🔬 Justificación Técnica

  • Modelo FULL obligatorio para poder hacer backup de logs transaccionales.
  • WITH INIT sobrescribe el archivo (útil para jobs nocturnos únicos).
  • Archivos .trn para logs permiten restauración a cualquier minuto.

💡 Buenas Prácticas

  • Crear 3 jobs en SQL Agent: uno por cada tipo de backup.
  • Alertar por correo si algún job falla.
  • Almacenar logs en subcarpeta con la fecha del día (ej: C:\Backups\Log\2024-11\).

Estrategia de Respaldo para Alta Disponibilidad

▼

📋 Enunciado

Diseñar una estrategia de recuperación ante desastres para la plataforma institucional de grados y títulos, garantizando recuperación máxima de 15 minutos.

💻 Script T-SQL

-- Estrategia: FULL dominical + DIFF diario + LOG cada 15 min

-- Verificar cadena de backups existente
SELECT
    CASE type
        WHEN 'D' THEN 'Full'
        WHEN 'I' THEN 'Diferencial'
        WHEN 'L' THEN 'Log'
    END AS TipoBackup,
    backup_start_date,
    backup_finish_date,
    backup_size/1024/1024 AS size_mb,
    compressed_backup_size/1024/1024 AS compressed_mb
FROM   msdb.dbo.backupset
WHERE  database_name = 'GradosTitulos'
  AND  backup_start_date >= DATEADD(DAY, -7, GETDATE())
ORDER BY backup_start_date DESC;

-- Log con RPO de 15 minutos
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\GT_Log_15min.trn'
    WITH COMPRESSION, CHECKSUM, NOINIT;

🔬 Justificación Técnica

  • msdb.dbo.backupset es la fuente oficial del historial de backups de SQL Server.
  • Monitorear compressed_mb vs backup_size valida la efectividad de la compresión.
  • NOINIT permite adjuntar múltiples logs al mismo archivo (.trn).

💡 Buenas Prácticas

  • Definir SLA documentado: RPO = 15 min, RTO = 30 min.
  • Realizar simulacros de recuperación trimestralmente.
  • Probar la cadena completa Full→Diff→Log en ambiente de pruebas.

Estrategia Corporativa de Respaldo (pérdida máx. 15 min)

▼

📋 Enunciado

Diseñar solución que incluya: respaldo completo dominical, diferencial diario, log cada 15 min, compresión y verificación automática de integridad.

💻 Script T-SQL

-- Procedimiento maestro de respaldo corporativo
CREATE OR ALTER PROCEDURE usp_BackupCorporativo
    @TipoBackup CHAR(4)  -- 'FULL','DIFF','LOG'
AS
BEGIN
    DECLARE @ruta NVARCHAR(500)
    DECLARE @ts   NVARCHAR(20) =
        CONVERT(NVARCHAR,GETDATE(),112) + '_' +
        REPLACE(CONVERT(NVARCHAR,GETDATE(),108),':','')

    IF @TipoBackup = 'FULL'
        SET @ruta = 'C:\Backups\Full\GT_Full_'+@ts+'.bak'
    ELSE IF @TipoBackup = 'DIFF'
        SET @ruta = 'C:\Backups\Diff\GT_Diff_'+@ts+'.bak'
    ELSE
        SET @ruta = 'C:\Backups\Log\GT_Log_'+@ts+'.trn'

    -- Ejecutar backup según tipo
    IF @TipoBackup IN ('FULL','DIFF')
        EXEC('BACKUP DATABASE GradosTitulos TO DISK='''+@ruta+''' WITH COMPRESSION,CHECKSUM'
           + CASE WHEN @TipoBackup='DIFF' THEN ',DIFFERENTIAL' ELSE '' END)
    ELSE
        EXEC('BACKUP LOG GradosTitulos TO DISK='''+@ruta+''' WITH COMPRESSION,CHECKSUM')

    -- Verificar integridad automáticamente
    RESTORE VERIFYONLY FROM DISK=@ruta WITH CHECKSUM

    -- Log de control
    INSERT INTO dbo.HistorialBackup(TipoBackup,FechaEjecucion,RutaArchivo,Estado)
    VALUES(@TipoBackup,GETDATE(),@ruta,'OK')
END

🔬 Justificación Técnica

  • Procedimiento único para los 3 tipos de backup reduce el riesgo de error humano.
  • RESTORE VERIFYONLY integrado garantiza que cada backup es restaurable.

💡 Buenas Prácticas

  • Llamar al procedimiento desde 3 SQL Agent Jobs diferentes con sus frecuencias.
  • Enviar alerta por correo si el INSERT en HistorialBackup no ocurre en el tiempo esperado.

Estrategia de Respaldo para Disaster Recovery

▼

📋 Enunciado

La Oficina de TI requiere un Disaster Recovery para GradosTitulos: reconstruir la BD completamente en un servidor alterno ante la pérdida total del servidor principal.

💻 Script T-SQL

-- Copiar backups a servidor alterno (xp_cmdshell o robocopy en job)
-- En servidor ALTERNO: restaurar la cadena completa

-- Paso 1: Restaurar FULL sin recuperar
RESTORE DATABASE GradosTitulos
    FROM DISK = '\\NAS\Backups\Full\GT_Full_Semanal.bak'
    WITH NORECOVERY,
    MOVE 'GradosTitulos'     TO 'D:\Data\GT.mdf',
    MOVE 'GradosTitulos_log' TO 'E:\Log\GT_log.ldf',
    REPLACE;

-- Paso 2: Aplicar DIFERENCIAL más reciente
RESTORE DATABASE GradosTitulos
    FROM DISK = '\\NAS\Backups\Diff\GT_Diff_Ayer.bak'
    WITH NORECOVERY;

-- Paso 3: Aplicar todos los LOGS en orden
RESTORE LOG GradosTitulos
    FROM DISK = '\\NAS\Backups\Log\GT_Log_0000.trn'
    WITH NORECOVERY;

RESTORE LOG GradosTitulos
    FROM DISK = '\\NAS\Backups\Log\GT_Log_0030.trn'
    WITH NORECOVERY;

-- Paso 4: Poner la BD en línea
RESTORE DATABASE GradosTitulos
    WITH RECOVERY;

-- Verificar estado
SELECT name, state_desc, recovery_model_desc
FROM   sys.databases
WHERE  name = 'GradosTitulos';

🔬 Justificación Técnica

  • NORECOVERY mantiene la BD en modo restoring para aplicar más backups.
  • MOVE redirige los archivos físicos a las rutas del servidor alterno.
  • REPLACE permite sobrescribir si la BD ya existía en el servidor destino.
  • El último RESTORE con RECOVERY pone la BD en línea.

💡 Buenas Prácticas

  • Documentar el runbook de DR con cada paso y tiempo estimado.
  • Realizar drill de DR semestralmente en el servidor alterno.
  • Mantener los backups en una ubicación accesible desde el servidor alterno (NAS o nube).

Restaurar después de eliminación accidental de registros

▼

📋 Enunciado

Un administrador eliminó accidentalmente registros relacionados con expedientes de titulación. Restaurar la base usando el respaldo más reciente.

💻 Script T-SQL

-- Primero: tail-log backup para capturar transacciones no respaldadas
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\GT_TailLog_Emergencia.trn'
    WITH NORECOVERY, NO_TRUNCATE, CHECKSUM;

-- Restaurar FULL
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Full\GT_Full_Ultimo.bak'
    WITH NORECOVERY, REPLACE;

-- Restaurar DIFERENCIAL
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Diff\GT_Diff_Ayer.bak'
    WITH NORECOVERY;

-- Restaurar LOGS hasta el momento del incidente
RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_Log_0000.trn'
    WITH NORECOVERY;

-- Restaurar tail-log (captura final)
RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_TailLog_Emergencia.trn'
    WITH RECOVERY;  -- ← pone online la BD

🔬 Justificación Técnica

  • El tail-log backup captura transacciones del log activo antes de restaurar.
  • NO_TRUNCATE preserva el log aunque la BD esté dañada.
  • Sin el tail-log se pierden las transacciones desde el último log backup.

💡 Buenas Prácticas

  • SIEMPRE hacer tail-log backup antes de restaurar (si la BD está online).
  • Practicar este procedimiento en ambiente de pruebas con datos reales copiados.

Recuperación después de Corrupción del Archivo MDF

▼

📋 Enunciado

El archivo principal de datos presenta corrupción física. Implementar el procedimiento de restauración completo utilizando respaldos disponibles.

💻 Script T-SQL

-- Diagnosticar corrupción
DBCC CHECKDB(GradosTitulos) WITH NO_INFOMSGS, ALL_ERRORMSGS;

-- Poner en modo EMERGENCY para leer datos
ALTER DATABASE GradosTitulos SET EMERGENCY;
ALTER DATABASE GradosTitulos SET SINGLE_USER;

-- Intentar reparación (ÚLTIMO RECURSO - puede haber pérdida)
DBCC CHECKDB(GradosTitulos, REPAIR_ALLOW_DATA_LOSS)
    WITH ALL_ERRORMSGS;

-- Si la reparación no es viable: restaurar desde backup
ALTER DATABASE GradosTitulos SET MULTI_USER;

-- Tail-log si el log no está corrupto
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\GT_TailLog_Corrupcion.trn'
    WITH NORECOVERY, NO_TRUNCATE;

-- Restaurar cadena completa
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Full\GT_Full_Ultimo.bak'
    WITH NORECOVERY, REPLACE,
    MOVE 'GradosTitulos'     TO 'C:\Data\GradosTitulos.mdf',
    MOVE 'GradosTitulos_log' TO 'C:\Log\GradosTitulos_log.ldf';

RESTORE DATABASE GradosTitulos WITH RECOVERY;

🔬 Justificación Técnica

  • DBCC CHECKDB es la herramienta oficial de SQL Server para detectar corrupción.
  • Modo EMERGENCY permite SELECT de datos aunque la BD esté dañada.
  • REPAIR_ALLOW_DATA_LOSS solo como último recurso (puede perder datos).

💡 Buenas Prácticas

  • Ejecutar DBCC CHECKDB semanalmente de forma programada.
  • Restaurar desde backup siempre que sea posible (no usar REPAIR).
  • Monitorear el Event Viewer del SO para señales de fallo de disco anticipado.

Verificación de Integridad de Backups

▼

📋 Enunciado

Verificar que todos los respaldos generados para GradosTitulos sean recuperables.

💻 Script T-SQL

-- Verificar backup sin restaurar
RESTORE VERIFYONLY
    FROM DISK = 'C:\Backups\Full\GT_Full_Ultimo.bak'
    WITH CHECKSUM;

-- Verificar y obtener información del backup
RESTORE HEADERONLY
    FROM DISK = 'C:\Backups\Full\GT_Full_Ultimo.bak';

-- Listar todos los backups NO verificados del último día
SELECT
    bs.database_name,
    bs.backup_start_date,
    bs.backup_finish_date,
    CASE bs.type
        WHEN 'D' THEN 'Full'
        WHEN 'I' THEN 'Diferencial'
        WHEN 'L' THEN 'Log'
    END AS TipoBackup,
    bmf.physical_device_name AS archivo,
    bs.is_damaged
FROM   msdb.dbo.backupset bs
JOIN   msdb.dbo.backupmediafamily bmf
    ON bs.media_set_id = bmf.media_set_id
WHERE  bs.database_name = 'GradosTitulos'
  AND  bs.backup_finish_date >= DATEADD(DAY,-1,GETDATE())
ORDER BY bs.backup_start_date DESC;

🔬 Justificación Técnica

  • is_damaged = 1 indica que SQL Server detectó corrupción al escribir el backup.
  • RESTORE HEADERONLY muestra metadatos sin leer todos los datos del backup.
  • backupmediafamily contiene la ruta física del archivo de backup.

💡 Buenas Prácticas

  • Nunca confiar en un backup que no ha sido verificado con VERIFYONLY.
  • Automatizar la verificación como paso final de cada job de backup.

Inventario Histórico de Copias de Seguridad (último mes)

▼

📋 Enunciado

Construir un reporte que muestre todos los respaldos realizados durante el último mes.

💻 Script T-SQL

-- Reporte completo del último mes
SELECT
    CASE bs.type
        WHEN 'D' THEN '🗂 Full'
        WHEN 'I' THEN '📋 Diferencial'
        WHEN 'L' THEN '📝 Log'
    END                              AS Tipo,
    CONVERT(VARCHAR,bs.backup_start_date,103)  AS Fecha,
    CONVERT(VARCHAR,bs.backup_start_date,108)  AS Hora,
    DATEDIFF(MINUTE,bs.backup_start_date,
             bs.backup_finish_date)              AS DuracionMin,
    ROUND(bs.backup_size/1024.0/1024,2)         AS TamañoMB,
    ROUND(bs.compressed_backup_size/1024.0/1024,2) AS ComprimidoMB,
    bmf.physical_device_name                    AS Archivo
FROM   msdb.dbo.backupset bs
JOIN   msdb.dbo.backupmediafamily bmf
    ON bs.media_set_id = bmf.media_set_id
WHERE  bs.database_name = 'GradosTitulos'
  AND  bs.backup_finish_date >=
       DATEADD(MONTH,-1,GETDATE())
ORDER BY bs.backup_start_date DESC;

🔬 Justificación Técnica

  • El ratio TamañoMB / ComprimidoMB permite evaluar la efectividad de la compresión.
  • DuracionMin permite detectar backups que tardaron más de lo habitual (posible problema).

💡 Buenas Prácticas

  • Exportar este reporte a Excel mensualmente para la auditoría de TI.
  • Comparar tamaños entre semanas para detectar crecimiento anormal.

Detección de Respaldos Obsoletos

▼

📋 Enunciado

Identificar copias de seguridad que excedan la política institucional de retención.

💻 Script T-SQL

-- Backups fuera de política de retención (más de 90 días)
SELECT
    bs.database_name,
    CASE bs.type
        WHEN 'D' THEN 'Full'
        WHEN 'I' THEN 'Diferencial'
        WHEN 'L' THEN 'Log'
    END                            AS TipoBackup,
    bs.backup_finish_date          AS FechaBackup,
    DATEDIFF(DAY,bs.backup_finish_date,GETDATE()) AS DiasAntiguedad,
    bmf.physical_device_name       AS ArchivoFisico,
    ROUND(bs.backup_size/1024.0/1024,2) AS TamañoMB
FROM   msdb.dbo.backupset bs
JOIN   msdb.dbo.backupmediafamily bmf
    ON bs.media_set_id = bmf.media_set_id
WHERE  bs.database_name = 'GradosTitulos'
  AND  bs.backup_finish_date < DATEADD(DAY,-90,GETDATE())
ORDER BY bs.backup_finish_date;

-- Limpiar registros de msdb (no borra archivos físicos)
EXEC msdb.dbo.sp_delete_backuphistory
    @oldest_date = DATEADD(DAY,-90,GETDATE());

🔬 Justificación Técnica

  • sp_delete_backuphistory limpia solo el catálogo en msdb, no los archivos físicos.
  • Los archivos .bak/.trn deben eliminarse por separado (script PowerShell o Maintenance Plan).

💡 Buenas Prácticas

  • Nunca eliminar backups manualmente sin confirmación de que existen backups más recientes válidos.
  • Usar Maintenance Plan "Cleanup Task" para borrar archivos .bak/.trn automáticamente.

Respaldo Manual de GradosTitulos con T-SQL

▼

📋 Enunciado

Realizar un respaldo completo usando únicamente instrucciones T-SQL.

💻 Script T-SQL

-- Backup completo manual
BACKUP DATABASE GradosTitulos
    TO DISK = 'C:\Backups\GT_Manual_'
           + CONVERT(VARCHAR,GETDATE(),112)+'.bak'
    WITH
        COMPRESSION,
        CHECKSUM,
        INIT,
        NAME = 'GradosTitulos Backup Manual',
        DESCRIPTION = 'Backup generado manualmente por DBA',
        STATS = 10;  -- Muestra progreso cada 10%

-- Confirmar que el backup fue registrado
SELECT TOP 1
    backup_size/1024/1024 AS TamañoMB,
    compressed_backup_size/1024/1024 AS ComprimidoMB,
    backup_finish_date
FROM   msdb.dbo.backupset
WHERE  database_name = 'GradosTitulos'
ORDER BY backup_finish_date DESC;

🔬 Justificación Técnica

  • STATS = 10 muestra el avance del backup en porcentaje en mensajes de SSMS.
  • INIT sobrescribe el archivo si ya existe (útil para backup diario con nombre fijo).

💡 Buenas Prácticas

  • Agregar NAME y DESCRIPTION para identificar fácilmente el backup.
  • Verificar con RESTORE HEADERONLY que el archivo fue creado correctamente.
5

Framework Automatizado de Respaldo Institucional

▼

📋 Enunciado

Desarrollar un procedimiento almacenado reutilizable que genere respaldos completos de GradosTitulos con fecha y hora en el nombre del archivo.

💻 Script T-SQL

-- Framework completo de respaldo automatizado
CREATE OR ALTER PROCEDURE usp_FrameworkBackup
    @BaseDatos  SYSNAME      = 'GradosTitulos',
    @TipoBackup VARCHAR(4)  = 'FULL',
    @RutaBase   NVARCHAR(256) = 'C:\Backups\'
AS
BEGIN
    SET NOCOUNT ON

    DECLARE @ts     VARCHAR(20)
    DECLARE @ext    VARCHAR(4)
    DECLARE @subdir VARCHAR(10)
    DECLARE @ruta   NVARCHAR(500)
    DECLARE @sql    NVARCHAR(MAX)

    SET @ts = CONVERT(VARCHAR,GETDATE(),112)
            + REPLACE(CONVERT(VARCHAR,GETDATE(),108),':','')

    SELECT
        @ext    = CASE @TipoBackup WHEN 'LOG' THEN '.trn' ELSE '.bak' END,
        @subdir = CASE @TipoBackup
                      WHEN 'FULL' THEN 'Full\'
                      WHEN 'DIFF' THEN 'Diff\'
                      ELSE 'Log\'
                  END

    SET @ruta = @RutaBase + @subdir
              + @BaseDatos + '_' + @TipoBackup
              + '_' + @ts + @ext

    SET @sql = CASE @TipoBackup
        WHEN 'LOG'
        THEN 'BACKUP LOG '+@BaseDatos+' TO DISK='''+@ruta+''' WITH COMPRESSION,CHECKSUM'
        ELSE 'BACKUP DATABASE '+@BaseDatos+' TO DISK='''+@ruta+''' WITH COMPRESSION,CHECKSUM'
           + CASE WHEN @TipoBackup='DIFF' THEN ',DIFFERENTIAL' ELSE '' END
    END

    EXEC sp_executesql @sql
    EXEC('RESTORE VERIFYONLY FROM DISK='''+@ruta+''' WITH CHECKSUM')

    PRINT 'OK: ' + @ruta
END

🔬 Justificación Técnica

  • Parámetros por defecto simplifican la llamada al SP desde cualquier job.
  • sp_executesql construye y ejecuta SQL dinámico de forma segura.
  • La verificación automática integrada elimina el riesgo de backups corruptos.

💡 Buenas Prácticas

  • Llamar al SP desde jobs de SQL Agent: EXEC usp_FrameworkBackup 'GradosTitulos','FULL'
  • Agregar manejo de errores con TRY/CATCH y registro en tabla de log.

Plan de Mantenimiento Semanal

▼

📋 Enunciado

Creación de un Plan de Mantenimiento Semanal para GradosTitulos.

💻 Script T-SQL

-- Plan semanal: Backup + Índices + Estadísticas + Integridad

-- 1. Verificación de integridad
DBCC CHECKDB(GradosTitulos) WITH NO_INFOMSGS;

-- 2. Reorganizar índices fragmentados (< 30%)
SELECT
    OBJECT_NAME(ips.object_id) AS Tabla,
    i.name                     AS Indice,
    ips.avg_fragmentation_in_percent AS Fragmentacion
FROM   sys.dm_db_index_physical_stats(
    DB_ID('GradosTitulos'), NULL, NULL, NULL, 'SAMPLED') ips
JOIN   sys.indexes i
    ON ips.object_id = i.object_id
    AND ips.index_id = i.index_id
WHERE  ips.avg_fragmentation_in_percent > 5
ORDER BY ips.avg_fragmentation_in_percent DESC;

-- 3. Reconstruir índices con alta fragmentación
ALTER INDEX ALL ON Tramite.ExpedienteTitulacion REBUILD
    WITH (ONLINE = OFF, FILLFACTOR = 80);

-- 4. Actualizar estadísticas de todos los esquemas
UPDATE STATISTICS Tramite.ExpedienteTitulacion WITH FULLSCAN;
UPDATE STATISTICS Academico.Estudiante WITH FULLSCAN;

-- 5. Backup completo al final
EXEC usp_FrameworkBackup 'GradosTitulos', 'FULL';

🔬 Justificación Técnica

  • Fragmentación >30% → REBUILD; entre 5-30% → REORGANIZE (menos bloqueante).
  • FULLSCAN genera estadísticas más precisas (vs muestreo automático).
  • FILLFACTOR = 80 deja 20% libre en cada página para futuras inserciones.

💡 Buenas Prácticas

  • Ejecutar el plan de mantenimiento en ventana de baja actividad (madrugada del domingo).
  • Separar en 2 jobs: uno para índices/estadísticas (sábado) y otro para backup (domingo).

Política de Retención Automatizada (eliminar > 180 días)

▼

📋 Enunciado

Diseñar una política institucional que elimine automáticamente respaldos con más de 180 días de antigüedad.

💻 Script T-SQL

-- Procedimiento de limpieza de backups obsoletos
CREATE OR ALTER PROCEDURE usp_LimpiarBackupsObsoletos
    @DiasRetencion INT = 180
AS
BEGIN
    DECLARE @fechaCorte DATETIME =
        DATEADD(DAY, -@DiasRetencion, GETDATE())

    -- Mostrar qué se va a eliminar
    SELECT
        database_name, type, backup_finish_date,
        backup_size/1024/1024 AS TamañoMB
    FROM   msdb.dbo.backupset
    WHERE  database_name = 'GradosTitulos'
      AND  backup_finish_date < @fechaCorte;

    -- Limpiar catálogo de msdb
    EXEC msdb.dbo.sp_delete_backuphistory
        @oldest_date = @fechaCorte;

    -- Registrar limpieza
    INSERT INTO dbo.HistorialBackup
        (TipoBackup, FechaEjecucion, RutaArchivo, Estado)
    VALUES(
        'CLEANUP', GETDATE(),
        'Eliminados backups anteriores a: ' +
        CONVERT(VARCHAR,@fechaCorte,103),
        'OK')

    PRINT 'Limpieza completada para backups anteriores a: '
         + CONVERT(VARCHAR,@fechaCorte,103)
END

🔬 Justificación Técnica

  • sp_delete_backuphistory solo limpia msdb (catálogo), no los archivos físicos del disco.
  • Los archivos físicos deben eliminarse con xp_cmdshell, PowerShell o Maintenance Plan Cleanup.

💡 Buenas Prácticas

  • Nunca eliminar el último Full disponible aunque supere los 180 días.
  • Mantener al menos 1 Full + Diferenciales + Logs de los últimos 30 días siempre.
  • Documentar la política de retención en el manual de procedimientos de TI.

1

Recuperación Antes de una Eliminación Accidental (10:15 → 10:14)

▼

📋 Enunciado

Un operador eliminó registros de expedientes a las 10:15 a.m. Restaurar la base exactamente a las 10:14 a.m.

💻 Script T-SQL

-- Paso 1: Tail-log backup URGENTE (capturar log activo)
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\TailLog_10h15.trn'
    WITH NORECOVERY, NO_TRUNCATE;

-- Paso 2: Restaurar FULL
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Full\GT_Full_Hoy.bak'
    WITH NORECOVERY, REPLACE;

-- Paso 3: Restaurar DIFERENCIAL de esta mañana
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Diff\GT_Diff_Hoy.bak'
    WITH NORECOVERY;

-- Paso 4: Aplicar LOG hasta las 10:14:59
RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_Log_Hoy.trn'
    WITH NORECOVERY,
    STOPAT = '2024-11-15 10:14:59';  -- ← 1 minuto antes del incidente

-- Paso 5: Poner en línea
RESTORE DATABASE GradosTitulos
    WITH RECOVERY;

-- Verificar que los registros fueron recuperados
SELECT COUNT(*) AS TotalExpedientes
FROM   Tramite.ExpedienteTitulacion;

🔬 Justificación Técnica

  • STOPAT aplica todas las transacciones hasta el timestamp indicado y omite las posteriores.
  • La precisión depende de la frecuencia de los backups de log (ideal: cada 15 min).
  • NORECOVERY en cada paso mantiene la BD en estado de restauración.

💡 Buenas Prácticas

  • Practicar la restauración point-in-time en ambiente de QA mensualmente.
  • Anotar la hora exacta del incidente tan pronto como se detecte.
  • Si hay duda sobre la hora, usar STOPAT 1-2 minutos antes del incidente reportado.
4

Recuperación Forense Institucional

▼

📋 Enunciado

Se detectó una modificación maliciosa durante una ventana específica de tiempo. Restaurar GradosTitulos exactamente al momento anterior al incidente y documentar el procedimiento.

💻 Script T-SQL

-- Paso 1: Leer el log activo para identificar la operación
SELECT TOP 20
    [Current LSN], Operation, Context,
    [Transaction ID], [Begin Time],
    AllocUnitName
FROM   fn_dblog(NULL, NULL)
WHERE  Operation IN ('LOP_INSERT_ROWS','LOP_DELETE_ROWS','LOP_MODIFY_ROW')
  AND  AllocUnitName LIKE 'Documento%'
ORDER BY [Begin Time] DESC;

-- Paso 2: Tail-log backup de emergencia
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\TailLog_Forense.trn'
    WITH NORECOVERY, NO_TRUNCATE, CHECKSUM;

-- Paso 3: Restaurar cadena hasta ANTES del incidente
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Full\GT_Full_Ultimo.bak'
    WITH NORECOVERY, REPLACE;

RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_Log_Anteayer.trn'
    WITH NORECOVERY;

-- STOPAT: 5 minutos antes del incidente detectado
RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_Log_Ayer.trn'
    WITH NORECOVERY,
    STOPAT = '2024-11-14 23:55:00';

RESTORE DATABASE GradosTitulos
    WITH RECOVERY;

-- Documentar la recuperación
INSERT INTO dbo.HistorialBackup
    (TipoBackup,FechaEjecucion,RutaArchivo,Estado)
VALUES('RESTORE-PITR',GETDATE(),
       'Restauración forense - incidente 2024-11-14','OK');

🔬 Justificación Técnica

  • fn_dblog() permite leer el log de transacciones activo para identificar el LSN del incidente.
  • LOP_MODIFY_ROW identifica filas modificadas en el log activo.
  • Documentar en HistorialBackup crea evidencia del proceso forense.

💡 Buenas Prácticas

  • Involucrar al área de seguridad informática antes de restaurar en caso de ataque.
  • Preservar una copia del log de auditoría antes de la restauración como evidencia.
  • Generar un informe del incidente con: qué ocurrió, cuándo, quién y cómo se recuperó.

Recuperación Forense de Modificación Maliciosa

▼

📋 Enunciado

Se detectó una modificación no autorizada en las tablas del esquema Documento relacionadas con resoluciones y diplomas. Restaurar point-in-time al estado exacto antes de la alteración.

💻 Script T-SQL

-- Identificar transacciones sobre Documento.Resoluciones
SELECT
    [Current LSN], [Transaction ID],
    Operation, [Begin Time],
    AllocUnitName
FROM   fn_dblog(NULL, NULL)
WHERE  AllocUnitName IN (
    'Documento.Resoluciones.PK_Resoluciones',
    'Documento.Diplomas.PK_Diplomas')
ORDER BY [Begin Time] DESC;

-- Restaurar con STOPATMARK (si se usó BEGIN TRANSACTION ... WITH MARK)
-- O usar STOPAT con el timestamp del momento anterior al incidente

-- Tail-log de emergencia
BACKUP LOG GradosTitulos
    TO DISK = 'C:\Backups\Log\TailLog_ModMaliciosa.trn'
    WITH NORECOVERY, NO_TRUNCATE;

-- Cadena de restauración completa
RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Full\GT_Full_Reciente.bak'
    WITH NORECOVERY, REPLACE;

RESTORE DATABASE GradosTitulos
    FROM DISK = 'C:\Backups\Diff\GT_Diff_Reciente.bak'
    WITH NORECOVERY;

-- Aplicar logs hasta 1 minuto antes de la alteración
RESTORE LOG GradosTitulos
    FROM DISK = 'C:\Backups\Log\GT_Log_Reciente.trn'
    WITH NORECOVERY,
    STOPAT = '2024-11-15 11:44:59';

RESTORE DATABASE GradosTitulos WITH RECOVERY;

-- Validar integridad post-restauración
DBCC CHECKDB(GradosTitulos) WITH NO_INFOMSGS;

🔬 Justificación Técnica

  • fn_dblog filtra por AllocUnitName para identificar exactamente qué tablas fueron modificadas.
  • DBCC CHECKDB post-restauración garantiza la integridad estructural de la BD recuperada.
  • STOPAT 1 minuto antes del incidente asegura que no se aplica la transacción maliciosa.

💡 Buenas Prácticas

  • Bloquear inmediatamente la cuenta del usuario sospechoso antes de restaurar.
  • Conservar el archivo de auditoría como evidencia antes de cualquier restauración.
  • Coordinar con el área legal antes de proceder si hay sospecha de delito informático.
