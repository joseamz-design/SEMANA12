Tema 01

Estructura de Almacenamiento y Archivos de Datos

📄 Archivo Primario (.mdf)

Punto de entrada único de la BD. Contiene el catálogo del sistema y apunta a los demás archivos físicos.

📑 Archivo Secundario (.ndf)

Extensión opcional. Distribuye datos en distintos discos para optimizar I/O en lecturas/escrituras concurrentes.

📋 Log de Transacciones (.ldf)

Registra cada operación DML. Indispensable para Point-in-Time Recovery y replicación.

📦 Filegroups

Agrupaciones lógicas de archivos físicos. Permiten colocar tablas e índices en hardware específico.

⚙️ FILEGROWTH

Crecimiento automático del archivo. Siempre definir en MB (ej: 5MB) para evitar fragmentación física.

🔍 sys.database_files

Vista del sistema que expone nombre lógico, ruta física, tipo, grupo y tamaño actual de cada archivo.

📐 Arquitectura física de GradosTitulos
┌─────────────────────────────────────────────────────┐ │ INSTANCIA SQL SERVER │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ BASE DE DATOS: GradosTitulos │ │ │ │ │ │ │ │ PRIMARY (Filegroup) IdxGroup (FG) │ │ │ │ ┌──────────────────┐ ┌────────────────┐ │ │ │ │ │ GradosTitulos │ │ GradosTitulos │ │ │ │ │ │ _data.mdf │ │ _idx.ndf │ │ │ │ │ │ SIZE: 20 MB │ │ SIZE: 8 MB │ │ │ │ │ └──────────────────┘ └────────────────┘ │ │ │ │ │ │ │ │ DocumentosGroup (FG) LOG │ │ │ │ ┌──────────────────┐ ┌────────────────┐ │ │ │ │ │ GradosTitulos │ │ GradosTitulos │ │ │ │ │ │ _docdata.ndf │ │ _log.ldf │ │ │ │ │ │ SIZE: 10 MB │ │ SIZE: 6 MB │ │ │ │ │ └──────────────────┘ └────────────────┘ │ │ │ └─────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────┘
T-SQL 
-- Consultar distribución de archivos físicos
USE GradosTitulos; GO
SELECT
  df.name          AS NombreLogico,
  df.physical_name AS RutaFisica,
  df.type_desc     AS TipoArchivo,
  fg.name          AS GrupoArchivos,
  (df.size * 8) / 1024 AS Tamano_MB
FROM sys.database_files df
LEFT JOIN sys.filegroups fg
  ON df.data_space_id = fg.data_space_id;

Tema 02

Propiedades y Configuraciones de Bases de Datos

🌐 Collation

Modern_Spanish_CI_AS — soporta tildes, ñ y orden alfabético español. CI=case-insensitive, AS=accent-sensitive.

📊 Compat. Level

Nivel 160 (SQL 2022) activa Intelligent Query Processing (IQP): joins adaptativos, feedback de memoria y paralelismo.

🛡️ PAGE_VERIFY CHECKSUM

El motor calcula checksum al escribir y verifica al leer. Detecta corrupción de páginas en disco de forma inmediata.

📈 Auto Growth (MB)

Crecimiento fijo en MB evita eventos bloqueantes de expansión. Porcentaje genera archivos cada vez más grandes.

🔄 Flujo de verificación PAGE_VERIFY CHECKSUM
ESCRITURA LECTURA ───────── ─────── Datos → [ Calcular Checksum ] → [ Datos + Checksum ] ↓ ↓ Página guardada en disco [ Recalcular Checksum ] ↓ ¿Coincide? SÍ → OK ✓ NO → Error 824 ✗
T-SQL 
-- Ajustar collation y nivel de compatibilidad
ALTER DATABASE GradosTitulos
  COLLATE Modern_Spanish_CI_AS; GO
 
ALTER DATABASE GradosTitulos
  SET COMPATIBILITY_LEVEL = 160; GO
 
ALTER DATABASE GradosTitulos
  SET PAGE_VERIFY CHECKSUM; GO
 
-- Verificar configuración actual
SELECT name, collation_name, compatibility_level,
       page_verify_option_desc
FROM  sys.databases
WHERE name = 'GradosTitulos';

Tema 03

Tipos de Recuperación (Simple, Full, Bulk-Logged)

🟢 SIMPLE

Log truncado en cada CHECKPOINT automáticamente. Sin Point-in-Time Recovery. Ideal para desarrollo o BD de solo lectura.

🔵 FULL

Registra absolutamente todo. Permite restaurar al minuto exacto antes de un fallo. Requiere LOG BACKUPs periódicos.

🟠 BULK_LOGGED

Híbrido: minimiza el log en operaciones masivas (BULK INSERT). Mantiene Point-in-Time entre backups de log.

📅 Estrategia de backup recomendada (modelo FULL)
Lunes 02:00 Lunes 08:00 Lunes 14:00 Lunes 20:00 │ │ │ │ ┌────▼────┐ ┌────▼────┐ ┌────▼────┐ ┌────▼────┐ │ FULL │ │ LOG │ │ DIFF │ │ LOG │ │ BACKUP │─────▶│ BACKUP │───▶│ BACKUP │───▶│ BACKUP │ └─────────┘ └─────────┘ └─────────┘ └─────────┘ │ │ └──────────── Recovery Point ───────────────────┘ (cualquier minuto entre backups)
T-SQL 
-- Cambiar modelo de recuperación
ALTER DATABASE GradosTitulos SET RECOVERY SIMPLE;  -- desarrollo
ALTER DATABASE GradosTitulos SET RECOVERY FULL;    -- producción
GO
-- Backup completo
BACKUP DATABASE GradosTitulos
  TO DISK = 'D:\Backups\GT_Full.bak'
  WITH COMPRESSION, STATS = 10; GO
-- Backup diferencial
BACKUP DATABASE GradosTitulos
  TO DISK = 'D:\Backups\GT_Diff.bak'
  WITH DIFFERENTIAL, COMPRESSION; GO
-- Backup del Log
BACKUP LOG GradosTitulos
  TO DISK = 'D:\Backups\GT_Log.bak'
  WITH COMPRESSION;

Tema 04

Administración de Usuarios y Roles de Seguridad

🔐 Jerarquía de seguridad SQL Server

🔑 LOGIN de Servidor

Credencial a nivel de instancia. CREATE LOGIN ... WITH PASSWORD. Punto de entrada al motor SQL Server.

👤 USER de Base de Datos

CREATE USER ... FOR LOGIN. Vincula el login a una BD específica. Cada BD tiene su propio espacio de usuarios.

🎭 Roles Fijos del Sistema

db_owner, db_datareader, db_datawriter, db_ddladmin. Conjuntos predefinidos de permisos listos para asignar.

🛠️ Custom Roles

CREATE ROLE permite combinar permisos granulares según las necesidades exactas del negocio o del área institucional.

T-SQL 
-- Crear login + usuario + asignar rol
CREATE LOGIN VendedorQhatu
  WITH PASSWORD = 'Oper@Qt2026!',
  DEFAULT_DATABASE = GradosTitulos; GO
 
USE GradosTitulos; GO
CREATE USER OperadorTramite FOR LOGIN VendedorQhatu;
EXEC sp_addrolemember 'db_datawriter', 'OperadorTramite';

Tema 05

Asignación de Permisos y Políticas de Acceso

✅ GRANT

Otorga un permiso. GRANT SELECT ON Tabla TO Usuario. WITH GRANT OPTION permite que el usuario lo delegue a otros.

🚫 DENY

Bloquea explícitamente. Prevalece sobre cualquier GRANT, incluso los heredados por membresía a un rol.

↩️ REVOKE

Elimina un GRANT o DENY previo. No garantiza denegación total si existen permisos por rol aún activos.

📋 Columna-Nivel

GRANT SELECT ON Tabla(Col1,Col2) restringe la visibilidad a columnas específicas. Protección de datos PII.

⚖️ Precedencia de permisos
DENY > GRANT heredado por ROL > GRANT directo ════════════════════════════════════════════════════ Usuario sin GRANT ni DENY → Acceso DENEGADO (default) GRANT al rol → Acceso PERMITIDO DENY directo al usuario → Acceso DENEGADO (siempre gana) GRANT directo al usuario → Acceso PERMITIDO (si no hay DENY)
T-SQL 
-- GRANT de columnas específicas (privacidad)
GRANT SELECT ON Academico.Persona
  (IdPersona, Nombres, ApellidoPaterno, Correo)
  TO PracticanteSistemas; GO
 
-- DENY explícito sobre tabla sensible
DENY SELECT ON Tramite.Pago TO OperadorTramite; GO
 
-- EXECUTE sobre SP (sin acceso directo a tablas)
GRANT EXECUTE ON Tramite.SP_RegistrarTramite
  TO OperadorTramite;

Tema 06

Monitoreo con SQL Server Activity Monitor

🖥️ Activity Monitor

Interfaz gráfica de SSMS (F8). Muestra % CPU, I/O, esperas y procesos activos en tiempo real sin escribir código.

⚡ dm_exec_query_stats

DMV con estadísticas acumuladas de consultas en caché: CPU total, lecturas, tiempo de ejecución y conteo.

🔒 dm_exec_requests

Procesos activos en este instante. blocking_session_id > 0 identifica cadenas de bloqueo que generan timeouts.

📋 dm_db_index_usage_stats

Uso real de índices desde último reinicio. Detecta índices sin seeks ni scans que solo consumen espacio y ralentizan DML.

🔍 Flujo de diagnóstico de rendimiento
1. Activity Monitor → Identificar proceso/sesión problemático ↓ 2. sys.dm_exec_requests → Filtrar por blocking_session_id > 0 ↓ 3. sys.dm_exec_query_stats → Ordenar por total_worker_time DESC ↓ 4. sys.dm_exec_sql_text → Obtener texto SQL exacto del plan ↓ 5. Acción: Optimizar query / Agregar índice / Matar sesión bloqueada
T-SQL 
-- Top 5 consultas por CPU en GradosTitulos
SELECT TOP 5
  SUBSTRING(st.text,1,200)         AS SQL_Text,
  qs.total_worker_time / 1000      AS CPU_ms,
  qs.execution_count               AS Ejecuciones
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.dbid = DB_ID('GradosTitulos')
ORDER BY qs.total_worker_time DESC;

Tema 07

SQL Server Agent – Tareas Automáticas

🤖 Arquitectura de SQL Server Agent
┌─────────────────────────────────────────────────┐ │ SQL SERVER AGENT │ │ │ │ ┌─────────┐ ┌──────────┐ ┌─────────────┐ │ │ │ JOB │ │ SCHEDULE │ │ OPERATOR │ │ │ │ (Tarea) │◀──│(Cuándo?) │ │ (Email) │ │ │ └────┬────┘ └──────────┘ └──────┬──────┘ │ │ │ │ │ │ ┌────▼────┐ Notifica ───┘ │ │ │ STEPS │ (T-SQL, PS, SSIS...) │ │ │ Step 1 │─▶ Éxito → Step 2 │ │ │ Step 2 │─▶ Fallo → Notificar / Reintentar │ │ └─────────┘ │ │ │ │ Historial almacenado en: msdb.dbo.sysjobhistory│ └─────────────────────────────────────────────────┘

⚙️ Jobs

Unidad de automatización. Agrupa pasos (steps) que se ejecutan en secuencia con lógica condicional entre ellos.

🕐 Schedules

Define frecuencia: diario, semanal, mensual, o al inicio del Agente. Múltiples schedules por job si es necesario.

📧 Operators

Destinatarios de alertas: email, pager, net send. El job notifica al completar, fallar, o en ambos casos.

🔔 Alerts

Reaccionan a errores (severity >= 20), condiciones de rendimiento o eventos WMI. Pueden ejecutar un job automáticamente.

T-SQL · msdb 
USE msdb; GO
-- 1. Operador para notificaciones
EXEC sp_add_operator
  @name          = 'DBA_Alertas',
  @email_address = 'dba@upla.edu.pe',
  @enabled       = 1; GO
-- 2. Job de respaldo diario
EXEC sp_add_job @job_name = 'Respaldo_Diario_GT';
EXEC sp_add_jobstep
  @job_name  = 'Respaldo_Diario_GT',
  @step_name = 'Backup Full',
  @command   = 'BACKUP DATABASE GradosTitulos
    TO DISK=''D:\Backups\GT_Auto.bak''
    WITH COMPRESSION;';
EXEC sp_add_schedule
  @schedule_name      = 'Diario_2AM',
  @freq_type          = 4,
  @active_start_time  = 020000;
EXEC sp_attach_schedule
  @job_name      = 'Respaldo_Diario_GT',
  @schedule_name = 'Diario_2AM';

1

Creación y distribución de archivos físicos de la base GradosTitulos

01 · Enunciado

¿Qué se pide?

Crear GradosTitulos con archivo primario, secundario para documentos históricos y log, en rutas físicas dedicadas. Validar con consulta.

03 · Justificación

¿Por qué?

Separar filegroups aísla I/O de tablas operativas vs históricas, reduce contención y permite respaldo parcial independiente.

04 · Buenas Prácticas

Estándares aplicados

SET SINGLE_USER antes del DROP. FILEGROWTH en MB fijo. Nomenclatura de rutas con año (BaseDatos2026\).

Script · Proyecto 1 
USE master; GO
-- Limpiar si existe
IF EXISTS(SELECT 1 FROM sys.databases WHERE name=N'GradosTitulos')
BEGIN
  ALTER DATABASE GradosTitulos
    SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
  DROP DATABASE GradosTitulos;
END; GO
 
CREATE DATABASE GradosTitulos
ON PRIMARY (
  NAME = GradosTitulos_Data,
  FILENAME = 'D:\BaseDatos2026\GradosTitulos_data.mdf',
  SIZE=20MB, MAXSIZE=20GB, FILEGROWTH=5MB
),
FILEGROUP DocumentosGroup (
  NAME = GradosTitulos_DocData,
  FILENAME = 'D:\BaseDatos2026\GradosTitulos_docdata.ndf',
  SIZE=10MB, MAXSIZE=10GB, FILEGROWTH=2MB
)
LOG ON (
  NAME = GradosTitulos_Log,
  FILENAME = 'D:\BaseDatos2026\GradosTitulos_log.ldf',
  SIZE=6MB, MAXSIZE=2GB, FILEGROWTH=2MB
); GO
 
-- Verificar distribución de archivos
USE GradosTitulos; GO
SELECT df.name AS NombreLogico,
       df.physical_name AS RutaFisica,
       df.type_desc     AS TipoArchivo,
       fg.name          AS GrupoArchivos,
       (df.size*8)/1024 AS Tamano_MB
FROM   sys.database_files df
LEFT JOIN sys.filegroups fg
  ON  df.data_space_id = fg.data_space_id;
Ejercicios propuestos
1) Filegroup exclusivo para Índices (IdxGroup)
ALTER DATABASE GradosTitulos ADD FILEGROUP IdxGroup;
ALTER DATABASE GradosTitulos ADD FILE (
  NAME=GradosTitulos_Idx,
  FILENAME='D:\BaseDatos2026\GradosTitulos_idx.ndf',
  SIZE=8MB, MAXSIZE=5GB, FILEGROWTH=2MB
) TO FILEGROUP IdxGroup;
2) Espacio disponible por Filegroup
SELECT fg.name, SUM(df.size*8.0/1024) AS Total_MB,
  SUM(FILEPROPERTY(df.name,'SpaceUsed')*8.0/1024) AS Usado_MB
FROM sys.database_files df
LEFT JOIN sys.filegroups fg ON df.data_space_id=fg.data_space_id
GROUP BY fg.name;

Definición de modelo de recuperación y respaldo para GradosTitulos

01 · Enunciado

¿Qué se pide?

Cambiar secuencialmente a SIMPLE → BULK_LOGGED → FULL. Explicar impacto de cada modelo. Ejecutar backup completo con compresión.

03 · Justificación

¿Por qué FULL?

Producción académica requiere Point-in-Time Recovery. Un fallo durante inscripciones masivas debe recuperarse al minuto exacto previo al incidente.

04 · Buenas Prácticas

Estándares

Usar COMPRESSION en todos los backups. STATS=10 para seguimiento. Almacenar en ruta dedicada con fecha en el nombre del archivo.

Script · Proyecto 3 
USE master; GO
-- Recorrer modelos de recuperación
ALTER DATABASE GradosTitulos SET RECOVERY SIMPLE;
GO    -- Log truncado en CHECKPOINT, sin PiTR
ALTER DATABASE GradosTitulos SET RECOVERY BULK_LOGGED;
GO    -- Cargas masivas mínimamente logueadas
ALTER DATABASE GradosTitulos SET RECOVERY FULL;
GO    -- Máxima protección, PiTR habilitado
 
-- Backup completo (base de toda estrategia)
BACKUP DATABASE GradosTitulos
  TO DISK = 'D:\Backups\GradosTitulos_Full_20260628.bak'
  WITH COMPRESSION, STATS=10,
       NAME='Full Backup GradosTitulos 2026-06-28';
GO
-- Verificar modelo final
SELECT name, recovery_model_desc
FROM  sys.databases WHERE name='GradosTitulos';
Ejercicios propuestos
1) Respaldo diferencial

2) Respaldo del Log de Transacciones

Identificación y solución de procesos lentos y bloqueos

01 · Enunciado

¿Qué se pide?

Describir KPIs del Activity Monitor. Desarrollar consultas T-SQL con DMVs para rastrear bloqueos y las 5 consultas de mayor consumo de CPU.

03 · Justificación

¿Por qué DMVs?

Las DMVs dan visibilidad en tiempo real sin herramientas externas. Son la base del diagnóstico sistemático de rendimiento en producción.

Script · Proyecto 6 
-- Top 5 consultas por consumo de CPU
SELECT TOP 5
  SUBSTRING(st.text,1,200)          AS SQL_Text,
  qs.total_worker_time/1000         AS CPU_ms,
  qs.execution_count                AS Ejecuciones,
  qs.total_elapsed_time/1000        AS TiempoTotal_ms
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.dbid = DB_ID('GradosTitulos')
ORDER BY qs.total_worker_time DESC;
GO
-- Bloqueos activos en tiempo real
SELECT
  r.session_id           AS Sesion,
  r.blocking_session_id  AS BloqueadaPor,
  r.wait_type, r.wait_time,
  SUBSTRING(st.text,1,150) AS ConsultaActual
FROM  sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
WHERE r.blocking_session_id > 0;
Ejercicios propuestos
1) Índices sin uso (candidatos a eliminar)
SELECT OBJECT_NAME(i.object_id) AS Tabla,
       i.name AS Indice, i.type_desc,
       ius.user_seeks+ius.user_scans+ius.user_lookups AS TotalUsos
FROM sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats ius
  ON i.object_id=ius.object_id AND i.index_id=ius.index_id
  AND ius.database_id=DB_ID()
WHERE i.type>0
  AND ISNULL(ius.user_seeks+ius.user_scans+ius.user_lookups,0)=0;

Automatización de respaldos y limpieza periódica con SQL Server Agent

01 · Enunciado

¿Qué se pide?

Job 1: respaldo completo diario a las 02:00 AM. Job 2: purga semanal (domingo 03:00) de registros de auditoría con más de 15 días.

03 · Justificación

Automatización

Elimina dependencia del factor humano en tareas críticas de mantenimiento. Garantiza ventanas de backup predecibles y auditables.

Script · Proyecto 7 
USE msdb; GO
-- Operador de notificaciones
EXEC sp_add_operator
  @name='DBA_Alertas', @enabled=1,
  @email_address='dba@upla.edu.pe'; GO
 
-- JOB 1: Respaldo diario
EXEC sp_add_job @job_name='Respaldo_Diario_GT';
EXEC sp_add_jobstep @job_name='Respaldo_Diario_GT',
  @step_name='Backup Full Comprimido',
  @command='BACKUP DATABASE GradosTitulos
    TO DISK=''D:\Backups\GT_Full_Auto.bak''
    WITH COMPRESSION, STATS=10;';
EXEC sp_add_schedule @schedule_name='Diario_2AM',
  @freq_type=4, @freq_interval=1, @active_start_time=020000;
EXEC sp_attach_schedule
  @job_name='Respaldo_Diario_GT', @schedule_name='Diario_2AM'; GO
 
-- JOB 2: Purga semanal de registros obsoletos

2) Reconstrucción automática de índices fragmentados (>30%)
DECLARE @obj NVARCHAR(256), @sql NVARCHAR(MAX);
DECLARE c CURSOR FOR
  SELECT OBJECT_NAME(object_id)
  FROM sys.dm_db_index_physical_stats
    (DB_ID('GradosTitulos'),NULL,NULL,NULL,'LIMITED')
  WHERE avg_fragmentation_in_percent>30 AND index_id>0;
OPEN c; FETCH NEXT FROM c INTO @obj;
WHILE @@FETCH_STATUS=0 BEGIN
  SET @sql='ALTER INDEX ALL ON '+@obj+' REBUILD';
  EXEC(@sql);
  FETCH NEXT FROM c INTO @obj;
END
CLOSE c; DEALLOCATE c;

Implementación de auditoría forense automatizada mediante Triggers

01 · Enunciado

¿Qué se pide?

Crear tabla AuditoriaPersona y trigger AFTER DELETE sobre Academico.Persona. Capturar: datos eliminados, SYSTEM_USER y SYSDATETIME().

03 · Justificación

Forense digital

La protección de datos personales (Ley 29733 Perú) obliga a registrar quién eliminó qué y cuándo. Los triggers garantizan trazabilidad sin modificar el código de aplicación.

Script · Proyecto 9 
USE GradosTitulos; GO
-- Tabla de auditoría forense
CREATE TABLE Academico.AuditoriaPersona (
  AuditId       INT IDENTITY(1,1) PRIMARY KEY,
  IdPersona     INT,
  NombresElim   NVARCHAR(200),
  DNIElim       VARCHAR(20),
  UsuarioSistema NVARCHAR(128),
  FechaHora     DATETIME2 DEFAULT SYSDATETIME()
); GO
 
-- Trigger AFTER DELETE: captura forense
CREATE TRIGGER TR_Auditoria_EliminarPersona
ON Academico.Persona AFTER DELETE
AS BEGIN
  SET NOCOUNT ON;
  INSERT INTO Academico.AuditoriaPersona
    (IdPersona, NombresElim, DNIElim, UsuarioSistema)
  SELECT d.IdPersona,
         d.Nombres+' '+d.ApellidoPaterno,
         d.DNI, SYSTEM_USER
  FROM deleted d;
END; GO
 
-- Trigger INSTEAD OF: bloqueo en fines de semana
CREATE TRIGGER TR_Seguridad_Horario_Docente
ON Academico.Docente INSTEAD OF DELETE, UPDATE
AS BEGIN
  IF DATEPART(WEEKDAY,GETDATE()) IN (1,7)
  BEGIN
    RAISERROR('Bloqueado fines de semana.',16,1);
    ROLLBACK;
  END
END;
Ejercicios propuestos
1) Trigger AFTER UPDATE para auditar cambios en Tramite.Pago
CREATE TRIGGER TR_Auditoria_ModificarPago
ON Tramite.Pago AFTER UPDATE
AS BEGIN
  SET NOCOUNT ON;
  INSERT INTO Tramite.AuditoriaPago
    (PagoId, MontoAnterior, MontoNuevo, Usuario)
  SELECT i.PagoId, d.Monto, i.Monto, SYSTEM_USER
  FROM inserted i INNER JOIN deleted d ON i.PagoId=d.PagoId
  WHERE i.Monto <> d.Monto;
END;

Simulación de recuperación ante contingencias de datos

01 · Enunciado

¿Qué se pide?

Simular borrado accidental del catálogo institucional. Restaurar completamente desde el .bak del Proyecto 3. Validar integridad con DBCC CHECKDB.

03 · Justificación

Plan de contingencia

Todo DBA debe poder ejecutar un restore completo bajo presión. Practicar el proceso en entorno controlado garantiza respuesta efectiva ante desastres reales.

04 · Buenas Prácticas

Estándares

SINGLE_USER antes del RESTORE. WITH REPLACE para sobrescribir. MULTI_USER al finalizar. DBCC CHECKDB para validar integridad post-restauración.

Script · Proyecto 10 
-- Simular el desastre
USE GradosTitulos; GO
DELETE FROM Catalogo.EstadoTramite; -- ERROR INTENCIONAL
GO
 
-- Restauración completa desde backup
USE master; GO
ALTER DATABASE GradosTitulos
  SET SINGLE_USER WITH ROLLBACK IMMEDIATE; GO
 
RESTORE DATABASE GradosTitulos
  FROM DISK = 'D:\Backups\GradosTitulos_Full_20260628.bak'
  WITH REPLACE, RECOVERY, STATS=10; GO
 
-- Volver a modo multiusuario
ALTER DATABASE GradosTitulos SET MULTI_USER; GO
 
-- Verificar datos restaurados
USE GradosTitulos; GO
SELECT COUNT(*) AS TotalEstados
FROM Catalogo.EstadoTramite; GO
 
-- Auditoría de integridad post-restauración
DBCC CHECKDB(GradosTitulos) WITH NO_INFOMSGS;
